# 게임 데이터 편집기 구조

`editor/server.py`는 한국어 최신 빌드나 일본어 원본을 읽고, 검증된 ROM
필드만 수정한 별도 ROM을 만듭니다. 출력은
`roms/builds/Langrisser II (Korean Editor Edit).md`이며 기준 ROM을
덮어쓰지 않습니다.

## 시나리오 배치

- 포인터 표: `0x18005E`
- 각 시나리오 헤더 `+0x0C`: 고정 배치 목록 포인터
- 레코드 크기: 36바이트
- 편집 필드: LV `+0x0E`, AT/DF `+0x12/+0x13`, 클래스 `+0x1B`,
  용병 6칸 `+0x1E..+0x23`
- 읽기 전용: 이름 ID, 좌표, 진영, 이벤트 플래그

좌표와 진영은 이벤트가 실행 중에 다시 쓸 수 있으므로 아직 편집하지
않습니다. 클래스 ID만 바꿔도 런타임 마법·소환 권한이 자동으로 붙지는
않습니다.

## 아이템

- 효과 표: `0x060530`
- 아이템당 8바이트: `(효과 종류, 값)` 4쌍
- 가격 표: `0x0A1D32`
- 가격 단위: ROM 값 1당 게임 표시 10P

일반 효과 종류는 AT, DF, MV, 지휘범위, A/D보정, 마법 사거리·대미지,
마법저항, 소환 ID입니다. 경험치 2배, 공격 사거리, 룬스톤 클래스 초기화,
MP 2배 같은 일부 아이템 동작은 개별 코드에 하드코딩되어 있어 현재
편집기에서는 읽기 전용 참고 정보로만 표시합니다.

아이콘은 `captures/run/97e0_item_id01.png`부터 `id37.png`까지의 검증된
320x240 상점 캡처에서 선택 행의 16x16 영역을 잘라 생성합니다.
선택 행은 X=24, `Y=42+16*((ID-1)%5)`입니다.
`tools/build_item_icon_assets.py`가 같은 좌표 계산과 37개 완전성을
검증합니다. 이전 14픽셀 계산은 아래쪽 아이템일수록 다음 아이콘과 섞이므로
사용하지 않습니다. 이 PNG는 에디터 표시 자산이며 ROM 그래픽을 수정하지
않습니다.

## 클래스 체인지

- 지휘관별 포인터 표: `0x08253A`
- 지휘관: 엘윈부터 제시카까지 10명
- 지휘관당 전이: 일반 9개와 최종 1개
- 일반 전이: 현재 클래스 1개와 후보 3개
- 최종 전이: 현재 클래스 1개와 후보 1개, 뒤에 `FFFF`

ROM의 10개 전이 레코드는 10단계가 아니다. 화면은 다음 소유 관계를
실제 최대 5단계 트리로 펼칩니다.

1. 레코드 0: 1단계 시작 클래스에서 2단계 후보 3개
2. 레코드 1..3: 각 2단계 클래스에서 3단계 후보
3. 레코드 4..8: 각 3단계 클래스에서 4단계 후보
4. 레코드 9: 지정된 4단계 클래스에서 숨은 5단계 후보

현재 클래스가 한 지휘관 경로 안에서 중복되거나 클래스 ID 범위를 벗어나면
빌드를 거부합니다.

### 전직 용병 해금

- 클래스 레코드 표: `0x05EDDC`
- 레코드 크기: `0x1C`
- 새 용병 비트 인덱스: `+0x1A`, `+0x1B`
- `FF`: 해금 없음
- 저장값 `00..0F`: 실제 용병 클래스 ID `62..71`

`0x014C68..0x014CAC`의 전직 적용 코드는 두 비트를 지휘관의 기존 고용
비트마스크에 추가합니다. 따라서 에디터의 두 칸은 그 클래스가 전직 시
**새로** 해금하는 용병이며, 전체 고용 목록을 덮어쓰지 않습니다.

### 클래스와 용병 그림

- 일반 클래스 스프라이트 ID 표: `0x05DDE6`, 157개 워드
- 지휘관별 스프라이트 포인터 표: `0x05DB80`, 10개 포인터
- 16x16 스프라이트 그래픽 시작: `0x052980`
- 스프라이트당 128바이트, 8x8 타일 4개

`tools/build_class_sprite_assets.py`는 일본판 ROM에서 일반 클래스 157종의
4개 원본 팔레트와 10명 지휘관의 클래스별 전용 그림을 추출합니다.
시나리오와 클래스 트리는 지휘관 전용 그림을 우선합니다. 지휘관 문맥이
없는 클래스 선택기는 일반 그림을 쓰되, 일반 스프라이트 표가 아니키
대체 그림 `0x18`을 가리키는 플레이어 클래스라면 ROM의 첫 지휘관
오버라이드를 대표 그림으로 사용합니다. 용병 선택기는 항상 일반 클래스
그림을 사용합니다. 클래스 체인지 화면의 다른 CRAM 행은 진영 팔레트가
아니므로 시나리오 진영과 무관하게 검증된 `p1`만 사용합니다. 이전
`class-previews`는 후보 화면의 용병 편성 영역을 자른 것이어서 캐릭터
클래스 그림으로 사용하지 않습니다. 외부 클래스 체인지 표와 이미지
모음은 이름·트리의 육안 대조에만 사용하고 ID 소유권은 ROM을 기준으로
결정합니다.

클래스 이름과 그림이 같아도 ID를 합치면 안 됩니다. 예를 들어 `2D`와
`2E`는 모두 일본 원본에서 파이터로 표시되고 일반 스프라이트 ID도 같지만,
28바이트 클래스 레코드는 `+0x0F`, `+0x10`, `+0x13`이 다릅니다.
고정 배치 전수 조사에서 `2D`는 시나리오 1·2·3·6의 일반 제국 지휘관
12명, `2E`는 시나리오 1의 발드 한 명만 사용합니다. 시나리오 LV/AT/DF는
클래스 레코드가 아니라 각 36바이트 배치 레코드에 별도로 저장됩니다.
나머지 클래스 레코드 필드는 의미와 적용 코드를 검증하기 전까지 편집기에
노출하지 않습니다. 에디터에서는 두 파이터를 육안으로 구분하기 위해
`2E` 대표 PNG에만 보라색 갑옷·빨간 방패 내부 팔레트를 적용합니다.
게임 빌드도 같은 구분을 반영합니다. 원본 `0x7E`의 두 애니메이션
프레임을 확장 ROM `0x2F0000`, `0x2F5900`에 복제해 색 인덱스만 바꾸고,
`2E` 표 항목만 전용 스프라이트 ID `0x53AD`로 연결합니다. 로더
`0x0110F4..0x011104`와 `0x011474..0x01148A`는 16비트 ID를
`ID << 7`로 계산하므로 이 확장 주소를 그대로 읽습니다. `2D`와 원본
스프라이트 `0x7E`는 수정하지 않습니다.

## 통합 빌드

`POST /api/build`는 한 번의 요청에서 다음 데이터를 순서대로 적용합니다.

1. 불러온 모든 시나리오 수정
2. 37개 아이템 가격과 효과
3. 10명 지휘관의 클래스 체인지 경로
4. 157개 클래스의 전직 용병 해금 두 칸
5. MD 헤더 체크섬 재계산

편집하지 않은 전체 데이터를 다시 전달한 회귀 검증에서는 결과 ROM이 기준
한국어 ROM과 바이트 단위로 동일했습니다. 현재 기준 체크섬은 `FDC9`,
SHA-256은
`7ad7d2199378380a6abb7bac94825799e57d7bf3f3122ded45225c7270a2b19c`
입니다.

## 실험 클래스 디자인 탭

`테스트 체인지`와 `AI 클래스`는 에디터 표시용 PNG만 읽는 비교 화면이다.
두 탭의 매니페스트에는 `rom_effect: none; preview PNG assets only`가
기록되어 있으며 `POST /api/build` 요청에도 이 데이터가 들어가지 않는다.
실제 게임에 적용된 별도 디자인은 시나리오 1 발드의 클래스 `2E`뿐이다.

두 실험 생성기는 지휘관별 클래스 그림 ID를 묶고 `(단계, 클래스 ID)`로
정렬한다. 같은 그림을 공유하는 묶음의 `group_rank == 0`은 ROM에서
추출한 16×16 PNG와 바이트 단위로 같아야 한다. 그 뒤 순위의 클래스만
변형하며 현재 전체 170개 항목 중 102개가 이 조건에 해당한다.

`tools/build_test_class_sprite_assets.py`는 원본의 검은 윤곽선과
피부·머리카락 주변 얼굴 마스크를 마지막 단계에 복원한다. 밝은색과
어두운색 두 덩어리로 모든 픽셀을 치환했던 첫 실험은 얼굴과 갑옷 경계를
뭉갰으므로 폐기했다.

`tools/build_ai_class_sprite_assets.py`는 원작 16×16 지휘관 스트립의
얼굴·머리카락을 변경 금지 정체성 레퍼런스로 사용해 만든 클래스별 보드를
입력으로 쓴다. 보드마다 5×3, 4×3, 4×2, 3×3의 고정 격자와 클래스 ID
순서를 코드에 기록해 눈대중
크롭으로 옆 셀을 자르는 일을 막는다. 각 셀에서는 중앙 전경 연결 요소를
골라 어두운 배경과 이웃 셀 조각을 제외한다.

이전 변환기는 전경 경계를 자른 뒤 가로와 세로를 각각 16픽셀에 맞춰
늘렸기 때문에 긴 창·날개·말이 있는 원화가 찌그러지고 논리 픽셀 격자가
어긋났다. 이후 15×15 안에 맞춘 결과도 한 줄을 불필요하게 잃었다.
현재 변환기는 종횡비를 유지하며 긴 축에 16픽셀 전체를 쓰고, 가로 중앙과
16번째 행의 바닥선에 정렬한다. 축소는 최근접 표본을 쓰고 희소색 보존에
유리한 최대 범위 15색 팔레트로 정리하므로 투명색을 포함해 최대 16색이다.
축소 결과 위에 ROM 얼굴·머리 사각형을 덮는 단계는 목, 어깨, 무기와
탈것을 잘라 이질적인 큰 머리 덩어리를 만들었으므로 제거했다. 대신 원작
16×16 지휘관을 정체성 레퍼런스로 삼아 생성한 원화의 얼굴·머리와 몸 연결을
그대로 축소해 캐릭터 일관성과 클래스 실루엣을 함께 유지한다. 사용 원화는
`docs/assets/ai-class-source/identity-locked-class-boards`, 결과는
`editor/static/ai-class-sprites`에 보존한다.

엘윈은 위 변환과 자동 재색칠 결과의 형체가 읽히지 않아 종전 10개 실험
시안을 제거했다. 현재 10개 항목 모두 ROM 원본을 표시한다. 새 검증은
`tools/pixellab_elwin_inpaint.py`가 실제 ROM 16×16을 입력으로 사용하고,
얼굴·머리·연결부를 검은 마스크로 생성 전부터 고정한 상태에서 몸과 장비만
같은 16×16 캔버스에 인페인트한다. 큰 원화를 축소하거나 생성 뒤 머리를
붙이는 단계는 없다. 잠금 픽셀이 하나라도 달라진 후보는 자동 탈락시키고,
나머지도 실제 크기와 최근접 확대본을 사람이 함께 확인한 뒤에만 적용한다.

초기 로드 인페인트는 아래쪽 손의 피부색 덩어리를 얼굴로 잘못 골라 몸통을
잠갔기 때문에 폐기했다. 이후 외부 투명 배경과 원본 실루엣 전체를 잠근
시드 1337은 형태 손실은 막았지만 30픽셀 재색칠에 그쳐 클래스 차이가
보이지 않았다. 이 후보도 폐기하고 ROM 원본으로 복원했다. 다음 프롬프트는
위쪽 얼굴·갈색 머리만 정체성 기준으로 유지하고, 몸 아래의 갑옷·망토·방패
실루엣은 원본과 명백히 달라지도록 별도 생성 대상으로 지정한다.
